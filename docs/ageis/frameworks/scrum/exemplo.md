# 🌐 Projeto: To-Do App (Web)

Você e sua equipe vão desenvolver uma aplicação simples de **lista de tarefas (To-Do App)** em 3 semanas.

---

## 👥 Papéis

* **Product Owner (PO)** → define funcionalidades (ex: cadastro, login, criar tarefas)
* **Scrum Master** → organiza o processo e remove bloqueios
* **Time de Desenvolvimento** → dividido em:

  * Front-end (interface)
  * Back-end (API)
  * Banco de dados

---

## 📋 Product Backlog (lista geral)

Funcionalidades do sistema:

* Cadastro de usuário
* Login
* Criar tarefa
* Listar tarefas
* Editar tarefa
* Deletar tarefa
* Marcar como concluída

---

## 🔁 Organização por Sprints

### 📅 Sprint 1 (Fundação do sistema)

**Objetivo:** deixar a base funcionando

**Sprint Backlog:**

* BE: criar API básica (ex: estrutura com endpoints)
* DB: modelar banco (tabela de usuários e tarefas)
* FE: criar layout inicial (tela de login/cadastro)

👉 **Entrega:** usuário consegue se cadastrar e logar

---

### 📅 Sprint 2 (Funcionalidades principais)

**Objetivo:** CRUD de tarefas

**Sprint Backlog:**

* BE: endpoints para criar, listar, editar e deletar tarefas
* DB: ajustar relacionamentos (usuário ↔ tarefas)
* FE: tela de listagem + formulário de criação

👉 **Entrega:** usuário já consegue gerenciar tarefas

---

### 📅 Sprint 3 (Melhorias e finalização)

**Objetivo:** refinamento e UX

**Sprint Backlog:**

* FE: melhorar interface (botões, feedback visual)
* BE: validações e segurança básica
* DB: otimizações (índices, ajustes)
* Geral: testes e correções

👉 **Entrega:** aplicação pronta para uso

---

## 🔄 Como a equipe trabalha no dia a dia

### 📅 Daily Scrum (15 min)

Cada membro fala:

* O que fez
* O que vai fazer
* Se tem impedimentos

Exemplo:

* FE: “Terminei a tela de login, hoje começo a lista de tarefas”
* BE: “API de login pronta, hoje faço criar tarefa”
* DB: “Estrutura pronta, ajustando relações”

---

## 🔗 Integração entre FE, BE e DB

* Back-end define os endpoints (ex: `/tasks`)
* Front-end consome esses endpoints
* Banco armazena os dados

💡 Tudo evolui **em conjunto a cada sprint**, não separado até o final.

---

## 🔍 Sprint Review

No fim de cada sprint:

* A equipe demonstra o que foi feito (ex: login funcionando)
* PO valida se atende ao esperado

---

## 🔄 Retrospectiva

A equipe analisa:

* O que deu certo (ex: boa divisão entre FE e BE)
* O que melhorar (ex: integração demorou muito)

---

## 💡 Resultado

* O sistema cresce **aos poucos, mas sempre funcionando**
* Problemas aparecem cedo (ex: integração quebrando)
* A equipe se adapta rapidamente
